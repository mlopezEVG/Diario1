    • 1. Introducción a GIT:

    1.1.1 ¿Qué es Git?:

    Git es un software de control de versiones, diseñado por Linus Torvalds. Permite a los desarrolladores rastrear y gestionar los cambios en el código fuente (control de versiones).

    1.1.2 Conceptos básicos:
  • Repositorio: Un lugar donde se almacena todo el código de un proyecto.
  • Commit:Un "punto de control" guardado con un mensaje que describe los cambios realizados.
  • Rama (Branch): Una línea de trabajo paralela donde puedes realizar cambios sin afectar la rama principal.
  • Fusión (Merge): Combina los cambios de diferentes ramas en una sola.
  • Clonar (Clone): Copiar un repositorio a tu máquina local.

    • 1.1.3 Buenas prácticas:
  • Haz commits frecuentes y con mensajes claros.
  • Usa ramas para trabajar en nuevas características.
  • Fusiona ramas regularmente para evitar conflictos.

    • 1.2. Configuración inicial

    1.2.1 ¿En qué consiste?:

    En esta sección se explica cómo realizar la configuración inicial de Git, lo cual es necesario para registrar los "commits" correctamente. Los comandos para configurar Git son 3:

  • git config --global user.name "Tu Nombre":Este comando establece el nombre del usuario que aparecerá en los commits. El "--global" indica que esta configuración se aplicará a todos los repositorios de Git en tu máquina."Tu Nombre" es el nombre que deseas usar. Puede ser tu nombre real o un alias.
  • git config --global user.email "tuemail@ejemplo.com": Este comando establece el correo electrónico del usuario que aparecerá en los commits. Al igual que el anterior, el "--global" aplica esta configuración a nivel global en tu máquina.
  • git config --global core.editor "nombre_editor": Configura el editor de texto por defecto que Git utilizará para abrir archivo de texto. (opcional).

    • 1.3. Comandos

    1.3.1 Comandos Básicos:

    En esta sección se explica cómo realizar la configuración inicial de Git, lo cual es necesario para registrar los "commits" correctamente. Los comandos para configurar Git son 3:

  • git help: Muestra la ayuda del comando que se indique.
  • git init: Inicializa el repositorio local.

    • Para subir cambios al repositorio remoto, debemos entender varios conceptos; el directorio de trabajo es donde están nuestros archivos. Con GIT INIT creamos nuestro staging y área local.

    -Para agregar archivos al área staging utilizaremos GIT ADD NOMBRE_ARCHIVO, para agregar un archivo en especifico, tambien podemos utilizar "." y añade todos los archivos.

    Enviamos nuestros archivos al repositorio local. Agregando GIT COMMIT -m "" que hace referencia al mensaje o nombre que utilizaremos para identificar al commit y además, registra el commit.

    -Por último y una vez realizados los commits, debemos simplemente enviarlos al repositorio remoto utilizando GIT PUSH.

    Problemas que me he encontrado: Ha habido veces que me he confundido y he puesto en el mensaje de un commit algo que no quería poner y busqué cómo cambiar el mensaje de un commit ya hecho. También, añadía al staging todos los ficheros que había creado, en lugar de, subir uno en concreto para hacer un commit de uno en específico.

    1.3.2 Empezando a funcionar:
    • git status: Muestra el estado de los archivos en el repositorio (cambios, archivos en staging, etc.).
    • git log y git log --oneline: Muestra el historial de commits, detallado o en resumen.

    Problemas que me he encontrado: Al principio confundía "git status" con "git log --oneline". También he tenido problemas al utilizar "git log" en ramas que contenían muchos commits, el historial se hacía ENORME. Ahí descubrí que "--oneline" es mejor cuando tienes muchos commits o un proyecto grande.

    • git add [archivo/patrón]: Añade archivos al área de staging.
    • git commit -m 'mensaje': Confirma los archivos en staging con un mensaje descriptivo.

    • git checkout [id_commit]: Cambia el directorio de trabajo a un estado específico del commit.

    Problemas que me he encontrado: Al principio confundía mucho cuando el profesor decía de hacer "x" número de commits. Creaba todos los ficheros de golpe y al añadirlos con "git add ." y despúes hacer el commit de todos los archivos, se hacía un commit de todos los ficheros creados, en lugar de, un commit por fichero creado. Ahí aprendí a añadir ficheros en específico: git add [nombrefichero]

  • git diff: Muestra diferencias entre archivos o entre commits específicos.

    • Problemas que me he encontrado: Al haberlo practicado una vez en clase, no sé exactamente cómo funciona. Pero imagino que cuando las diferencias o conflictos sean más extensos serán más dificil de resolver o combinar.

    • git remote: Administra conexiones a repositorios remotos.
    • git push y git push -u origin main: Envía los commits al servidor remoto, configurando la rama para empujar automáticamente.

    Problemas que me he encontrado: Una vez me confundí y hice push cuando no debeía haberlo hecho y aprendí a eliminar un push en la rama remota utilizando "--delete".

    • git clean y git clean -f: Elimina permanentemente archivos sin seguimiento.
    • git reset y git reset --hard: Restaura archivos a un estado anterior. --soft, --mixed, y --hard definen el nivel de restauración.

    Problemas que me he encontrado: Usar --hard eliminará los cambios sin posibilidad de recuperarlos. A la próxima debería mirar el impacto de cada opción antes de usar este comando.

    • git tag: Crea y muestra etiquetas para marcar commits específicos.
    • git revert: Deshace cambios de un commit, creando un nuevo commit de reversión.

    Problemas que me he encontrado: Si hay conflictos en el código al revertir, resolverlos puede ser complicado. Este comando es útil para mantener el historial, pero tenemos que revisar bien los cambios antes de ejecutar.

    TEMA 2:

    2.1. Trabajando con Ramas en Git.

    2.1.1 Creación y Gestión de Ramas:
    • git branch: Lista todas las ramas existentes.
    • git branch [nombre_nueva_rama]: Crea una nueva rama.
    • git branch -d [nombre_rama]: Elimina una rama.
    • git branch -m [antiguo_nombre_rama] [nuevo_nombre_rama]: Renombra una rama.

    Problemas que me he encontrado:Con las ramas he tenido bastantes problemas, por ejemplo, he añadido archivos a ramas a las que no debería por no estar posicionada en la rama correcta. Me he confundido al ponerles un nombre y tuve que buscar cómo cambiarlo.

    También hemos aprendido a movernos entre ramas:
    • git checkout [nombre_rama]: Cambia de una rama a otra.
    • git checkout -b [nombre_rama]: Crea una nueva rama y cambia a ella.
    • git checkout [ID_commit]: Cambia a un commit anterior.

    Problemas que me he encontrado: Al principio confundía la sintaxis de crear y cambiarse de rama. En lugar de poner "-b" ponía "-m" como cuando realizamos un commit. Ahí comprendí que la "-b" era de "branch".

    Y fusionarlas:
    • git merge [nombre_rama]: Fusiona una rama con la activa.
    • git merge --abort: Cancela la fusión en caso de conflicto.

    Problemas que me he encontrado:He fusionado ramas incorrectas, porque la fusión se realiza sobre la rama activa y no estaba posicionada en la correcta.

    2.1.2 Visualización de Commits y Logs:
    • git log: muestra el historial de commits.
    • git log -p: historial con cambios detallados.
    • git log --oneline: historial en una sola línea por commit.
    • git log --oneline --all: historial de todas las ramas.

    Problemas que me he encontrado:Cuando he tenido proyectos muy grandes he utilizado "git log" y me ha aparecido un historial muy extenso.Por ello, me parece que es más cómodo y útil utilizar "oneline".

    2.1.3 Resolución de conflictos, visualización y diferencias:
    • git diff [branch_origen] [branch_destino]: muestra las diferencias entre ramas.
    • git log --merge: muestra información de los commits que causan conflictos.

    TEMA 3:

    3.1. Clone, Pull y Fetch.

    3.1.1. Clonar y Actualizar Repositorios:

    Aprende los comandos básicos de Git para gestionar repositorios locales y remotos:

    • git clone [remote_repo] [directorio_local]:Copia un repositorio remoto en tu máquina local.
    • git clone --branch [tag] [remote_repo]Clona una versión específica del repositorio remoto.
    • git clone --branch --single-branch [branch_name] [remote_repo]:Clona una rama específica del repositorio remoto.

    Actualizar los repositorios locales con los cambios del servidor remoto:

    • git pull [servidor_remoto]:Fusiona los cambios del repositorio remoto en el directorio de trabajo local.
    • git pull [servidor_remoto] [rama]:Descarga una rama específica del servidor remoto.
    • git pull --verbose [servidor_remoto]:Muestra un proceso detallado del comando pull.

    Problemas que me he encontrado: Como pull es una fusión entre fetch y merge, al hacer pull me encontré de repente que en la consola se había abierto un terminal para hacer commits de confirmación de los cambios para el merge. No me esperaba eso y no supe qué tenía que hacer ya que no dejaba escribir. Más tarde, comprendí que había que hacer un commit. Tampoco sabía que para hacer pull debías añadir la referencia al repositorio remoto con "git remote add".

    Si quieres solo descargar los cambios sin fusionar:

    • git fetch [nombre_servidor_remoto]:Recupera datos del proyecto remoto sin fusionar con tu trabajo actual.
    • git checkout fetch_HEAD:Cambia a la rama con los cambios descargados mediante fetch.
    • git checkout -b [new_Branch]:Crea una nueva rama basada en los cambios descargados del remoto.
    • git merge [rama]:Fusiona cambios de una rama específica con la rama actual.
    • git diff [rama_local] [rama_remota_cambios]:Compara los cambios entre dos ramas.

    Problemas que me he encontrado:"git diff" puede llegar a ser complicado debido a la comparación de las dos ramas.
